Real Time Strategy (RTS) is a sub-genre of Strategy games, that are played simultaneously in real time. Which means there is no separation between decision making and active phase and both players play at the same time (they don't alternate).

Game is inspired by small implementation of an RTS game by Santiago Ontañón.

"Santiago Ontañón (2013) The Combinatorial Multi-Armed Bandit Problem and its Application to Real-Time Strategy Games, In AIIDE 2013. pp. 58 - 64".

Project's source code can be found here.


  The Goal

Conquer the arena, by destroying all of the enemy castles.

  Rules

The game is played on a map width tiles wide and height tiles high. Each tile can only be occupied by one entity at the time - entities are defined as non-empty tiles: units and buildings owned by each player, but also neutral tiles like mines or walls.

At the beginning of each turn players get a list of every entity on the map and information of their x and y position, owner identificator, entityType and health.

Each player starts with certain amount of resources, that can also be found on the map and need to be harvested. Those resources can be used for creation of different buildings and units.

Orders given by players have to be written according the rules and order syntax. Incorrect orders will result in immediate elimination of the player. If an order fits the criteria, but it's not possible to be executed, because of external reasons (like different unit blocking the way) - it will fail, without player's elimination. Every entity owned by players has only specific list of the orders it can execute. All actions can be done in all 8 directions. Entity is only limited by the free space, other entities, it's own reach and size of it's step.

Entities can only move into empty tiles or the tiles that will be emptied the same turn (for example, if occupied by different unit owned by the player, that will also move the same turn). Otherwise the order will fail to be executed, without eliminating the player.

Entity tiles can be sorted based on their owner and type:

Neutral tiles

Neutral tiles have owner identificator value equal to -1. There are three types of those entities in this version of the game:

  • Walls - they are the main physical obstacle in the arena. If not defined differently, they are unbreakable, which means their base health is equal -1.
  • Gold Mines - source of Gold. Their health value defines the amount of resource left to be mined. When depleted - it disappears from the arena.
  • Forests - source of Wood. Their health value defines the amount of resource left to be logged. When depleted - it disappears from the arena.

Buildings

Buildings are part of player tiles so their owner corresponds to the player's ID (player = 0, enemy = 1). In this variant of the game they can't move or attack, but they can train units in the range of their reach. Usually much more costly to be built than other entities.

  • Castles - pivotal building for each player. The player who doesn't have any castles left loses. They can train Workers.
  • Barracks - training ground for more advanced units. Can train units like Light, Heavy and Ranged troops.

Units

Units are the more dynamic player tiles akin to chess pieces. In this variant of the game all of them can move at the speed of step tiles per turn. They can also do all the given actions in the range of their reach.

  • Workers - most universal unit. Their main functionality is to build and harvest resources. They can attack, but they aren't very durable, strong or fast, their range is also only limited to one tile in this variant of the game.
  • Light - more advanced combat unit. They are more durable and stronger than regular Workers. Their main characteristic is that they can move two tiles per turn in this variant of the game.
  • Heavy - the tank unit. Strongest and most durable unit in this variant of the game. They are more costly to produce.
  • Ranged - weak, long range unit. Their main characteristic is long attack range of three tiles.

Game turn

Each turn players sent one line consisting of semi-colon separated orders. Then referee checks their validity and then commits them in priority order of: building > training > moving > harvesting > attacking.

Any movement that can't be done immediately is then pushed to the end of the "move" queue, this repeats until the queue is empty or no more units can move. After "moving" phase any remaining "building" or "training" orders that weren't possible before, will be considered once again.

Conflicting orders are executed in random order, if not possible to be executed at the same time. In case of two units attacking each other - both of them get damaged.

Then referee cleans any defeated entities and depleted resources off the arena and checks, if any of the end conditions are met.

  Game Input

Initialization input
First line: 3 integers seperated by a space:
  • width - width of the arena.
  • height - height of the arena.
  • startGold - the amount of Gold each player gets at the beginning of the game.

Next 6 lines: 7 integers seperated by a space defining each player entity type:
  • entityType - numerical value representing entity type.
  • maxHP - maximum health entity can have.
  • reach - the range in which entity can do all it's actions.
  • attack - the amount of damage entity does when attacking.
  • step - the amount of tiles entity can move in one turn.
  • costGold - the amount of Gold needed to create this entity.
  • costWood - the amount of Wood needed to create this entity.

Input for one game turn
First line: 1 integer N, which represents the number of entities on the arena.
Next N lines: 5 space-separated integers defining each entity on the arena:
  • x - x position of the entity.
  • y - y position of the entity.
  • owner - identificator for the owner of the entity:
    • -1: neutral.
    • 0: player.
    • 1: enemy.
  • entityType - numerical value representing entity type:
    • Neutral entities:
      • 1: wall.
      • 2: gold mine.
      • 3: forest.
    • Player and enemy entities:
      • 0: castle.
      • 1: barracks.
      • 2: worker.
      • 3: light.
      • 4: heavy.
      • 5: ranged.
  • health - entity's remaining health.

Output for one game turn

A single line of orders separated with semicolons.

Orders

When writing "building" and "training" orders - for target entity type both numerical and string values are acceptable (e.g. "CASTLE" or "0"). Some of the orders include pairs of coordinates: first pair always refers to the ordered entity and the other refers to the target entity or location.

BUILD X Y tX tY [entityType] - orders entity to construct a building at the target.
TRAIN X Y tX tY [entityType] - orders entity to train an unit at the target.
MOVE X Y tX tY - moves unit from one point to the other.
ATTACK X Y tX tY - orders the entity to attack the target.
HARVEST X Y tX tY - orders the entity to harvest resources at the target.
WAIT - used when no orders need to be sent out.
Entities

Each entityType has certain list of orders it can do within it's reach. Each entity can only do one order per turn.

  • CASTLE: entityType: 0, maxHP: 10, reach: 3, attack: 0, step: 0, costGold: 5, costWood: 5,
    TRAIN WORKER
  • BARRACKS: entityType: 1, maxHP: 4, reach: 2, attack: 0, step: 0, costGold: 0, costWood: 5,
    TRAIN LIGHT TRAIN HEAVY TRAIN RANGED
  • WORKER: entityType: 2, maxHP: 1, reach: 1, attack: 1, step: 1, costGold: 1, costWood: 0,
    BUILD CASTLE BUILD BARRACKS MOVE ATTACK HARVEST
  • LIGHT: entityType: 3, maxHP: 4, reach: 1, attack: 2, step: 2, costGold: 1, costWood: 1,
    MOVE ATTACK
  • HEAVY: entityType: 4, maxHP: 8, reach: 1, attack: 4, step: 1, costGold: 2, costWood: 1,
    MOVE ATTACK
  • RANGED: entityType: 5, maxHP: 1, reach: 3, attack: 1, step: 1, costGold: 1, costWood: 1,
    MOVE ATTACK
Constraints
0 ≤ X < width
0 ≤ Y < height
Response time for first turn ≤ 1 s
Response time for one turn ≤ 50 ms
Turn limit is 200.
